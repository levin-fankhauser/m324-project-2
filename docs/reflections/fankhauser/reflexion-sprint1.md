# Fragen

# Inhalt
## 1.1. Welche verschiedenen Themen behandelten wir bis jetzt?
Wir behandelten in diesem Modul bereits viele verschiedene Teilthemen. Dies waren zum Beispiel eine Einführung in das Thema DevOps und für was es verwendet wird. Dazu kamen dann in den folgenden Wochen Themen wie Projektplanung, GitHub Flow, Conventional Commits, Versionisierung von Releases, Formatierung, Linting und Github Workflows. Zum Beispiel kann man mit Conventional Commits und Release Please einfach ein ChangeLog generieren lassen, was die Dokumentation bei releases deutlich vereinfacht. Durch Formatierung und Linting wird der Code in der Qualität besser und man verhindert Komplikationen mit unterschiedlichen Codestils, was die Wartbarkeit verbessert und vor allem in einem grösseren Projekt einen grossen Vorteil bietet. GitHub ist hier ebenfalls ein wichtiger Punkt. Mit GitHub halten wir uns an Richtlinien, wie Github Flow indem wir mit Feature branches arbeiten und den main blockieren um ein "Chrüsimüsi" mit Code und versionierung zu verhindern. Dazu haben wir mit Github Projects eine gute Plattform um Epics und Issues festzulegen und so unsere Anforderung genau festhalten können. Dazu unterstützt dies den Arbeitsflow mit Github und Pull Requests sehr gut. Ein wichtiger Punkt sind natürlich auch die Pipelines (Workflows) welche Github bietet. Mit dieser können wir den release please bot laufen lassen, welcher für wie gesagt das changelog schreibt oder ein linter welche alle linting errors prüft und in der pull requests anmerkt um so den Code noch besser zu verbessern. Dazu kommt dann das ganze Deployment mit AWS, welches wir am Anfang kurz angeschaut haben und in Zukunft sicher noch mehr dazu lernen werden.

## 1.2. Was haben die Themen mit dem Begriff DevOps zu tun?
Die Themen die wir durchgenommen haben sind alle relevant für den Begriff DevOps. Sie erleichtern den Entwicklungsprozess und helfen bei der Versionisierung und Releases. Vieles davon sind automatisierungen, welche den ganzen Workflow vom entwickeln bis releasen deutlich erleichtern und den Menschen ablösen eine repetitive arbeit zu machen. Dazu kommt, dass ein workflow so bald er läuft weniger fehler machen kann als ein mensch. All diese Themen die oben beschrieben sind tragen dazu bei diesen prozess möglichst einfach und reibungslos zu machen. All das beschreibt den Begriff DevOps.

# Gruppendynamik
## 1.3. Wie war die Zusammenarbeit im Team?
Die zusammenarbeit im Team war grundsätzlich gut. Manchmal gab es das Problem, dass nicht alle etwas zu tun hatten. Dies haben wir meistens damit gelöst, dass wir dann eine Task mit pair programming zusammen gemacht haben, damit alle etwas vom thema mitnehmen konnten. Da wir in den ersten Wochen wenig "richtig" programmiert haben (features umgesetzt) und mehr konfigurationen gemacht haben, war es ein wenig schwer die arbeit untereinander aufzuteilen, da oft eine aufgabe vom anderen abhängig war und man warten musste. Dies war aber ab heute (WO4) weniger ein Problem, da wir nun begonnen haben features umzusetzen. Zwischenmenschlicht hat dafür alles sehr gut geklappt. Wir kennen uns recht gut und können offen über aufgaben und ideen diskutieren. Jetzt wo wir langsam in den "Flow" des Moduls gefunden haben funktioniert alles ein wenig reibungsloser.

## 1.4. Was ist uns besonders gut gelungen, worauf sind alle richtig stolz?
Was ich finde was bei uns gut verlaufen ist, ist zum einen die Planung. Wir konnte alle mitwirken und haben alle etwas vor Augen, was wir gerne umsetzten möchten. Bis heute haben wir noch nicht wirklich features umgesetzt die speziell sind oder wir sehr stolz sind aber der ganze Arbeits Flow den wir seit anfang gemacht haben war cool.
